# Plan 03 — Multilingual Tutor Upgrade Roadmap

## 문서 목적
이 문서는 현재의 일본어 중심 튜터 앱을 **단일 코어 + 다국어 패키지 구조**로 전환하기 위한 상세 작업 계획안이다.

이 문서는 사용자와 이미 합의된 **3단계 마스터 플랜**을 기준으로 정리한다.

- **1단계: 엔진 추상화** — 앱을 일본어 전용 앱에서 언어 선택 가능한 공통 엔진으로 전환
- **2단계: 일본어 + 영어 먼저 완성** — 공통 엔진 위에서 일본어/영어를 실제 usable flow로 완성
- **3단계: 나머지 언어 확장** — 한국어/중국어/프랑스어/독일어/스페인어/베트남어/태국어/아랍어를 언어 패키지 방식으로 확장

즉, 지금 당장 필요한 것은 1단계를 마무리하고 2단계를 한 번에 밀어붙일 수 있는 기준을 명확히 해두는 일이다.

---


## 실행 원칙 — 끊지 말고 한 번에 진행
- 1단계 마감 작업은 **구조 구현 완료 → 가능한 QA 수행 → 런타임 QA 위임 정리 → 2단계 계획 수립**까지를 한 흐름으로 본다.
- 중간에 “여기까지 했는데 계속할까요?” 식의 확인 요청은 원칙적으로 하지 않는다.
- 아래와 같은 **실제 블로커**가 있을 때만 멈춘다.
  - 비밀키/외부 권한이 없어 실행 자체가 불가능한 경우
  - 결제/인증/데이터 마이그레이션처럼 사용자 영향이 큰 파괴적 변경이 필요한 경우
  - 제품 방향이 갈리는 선택지 때문에 결과물이 크게 달라지는 경우
- 브라우저/배포 환경이 필요한 QA는 가능한 범위까지 먼저 수행하고, 남는 항목은 현재 작업 중인 에이전트가 후속 체크리스트로 정리한 뒤 다음 단계 산출물 작성으로 바로 넘어간다.
- 즉, **QA가 일부 보류되어도 계획/문서/다음 구현 준비는 멈추지 않는다.**

## 최종 제품 비전

### 핵심 개념
- 사용자는 앱 실행 후 **학습 언어(target language)** 를 선택한다.
- 필요 시 **설명 언어(explanation/support language)** 와 **UI 언어(ui language)** 를 별도로 선택한다.
- 동일한 캐릭터/시나리오/음성/복습 기능이 언어 설정에 따라 동적으로 바뀐다.
- 신규 언어 추가 시 앱 전체를 갈아엎지 않고, **언어 패키지 추가** 중심으로 확장한다.

### 목표 지원 언어
- Japanese
- English
- Korean
- Chinese
- Thai
- French
- German
- Spanish
- Arabic
- Vietnamese

---

## 전체 추진 전략 (실행 범위 중심)

### 이번 문서의 범위
- 문서는 1/2/3단계 전체 구조를 명시한다.
- 다만 세부 작업은 **1단계 마감 + 2단계 즉시 실행**에 초점을 둔다.
- 3단계는 확장 순서와 패키지화 원칙까지 정의한다.

---

# 1단계 — 엔진 추상화

## 목표
- 일본어 전용 하드코딩 제거
- 언어 설정을 앱의 1급 개념으로 승격
- UI/시나리오/프롬프트/TTS를 언어 독립형 구조로 재설계

## 완료 기준
- 코드베이스 어디에도 “일본어 전용 전제”가 핵심 흐름을 막지 않아야 한다.
- 앱이 최소한 내부적으로는 `targetLanguage`, `supportLanguage`, `uiLanguage`를 기준으로 동작해야 한다.
- 기존 일본어 기능이 깨지지 않아야 한다.

---

## 1-1. 제품/도메인 모델 재설계

### 작업 목표
기존 `audience` 중심 상태에 언어 축을 추가한다.

### 세부 작업
- 전역 상태에 다음 개념 추가
  - `targetLanguage`: 학습 대상 언어
  - `supportLanguage`: 설명/해설/번역 언어
  - `uiLanguage`: UI 표시 언어
  - `difficultyFramework`: CEFR/JLPT/HSK/TOPIK/custom
  - `difficultyLevel`: A1, N5, HSK1 등 실제 레벨 값
- 기존 `subtitleSettings` 재설계
  - `showSupportTranslation`
  - `showPronunciationGuide`
  - `romanizationMode`
- `audience`는 유지하되, 언어 개념과 분리
- 세션/복습/대화 기록에도 언어 메타데이터 저장

### 산출물
- 앱 상태 타입 개편안
- 언어 메타데이터 타입 정의
- 난이도 프레임워크 공통 규격 문서

### 리스크
- 기존 persisted store와 호환성 깨질 수 있음
- 마이그레이션 코드 필요

---

## 1-2. 언어 메타데이터 레이어 신설

### 작업 목표
언어별 설정을 하드코딩이 아니라 선언형 데이터로 관리한다.

### 세부 작업
- `client/src/constants/languages.ts` 또는 유사 파일 신설
- 언어별 메타데이터 정의
  - 코드 (`ja`, `en`, `ko`, `zh`, `th`, `fr`, `de`, `es`, `ar`, `vi`)
  - 표시명
  - 기본 설명 언어 후보
  - 기본 난이도 프레임워크
  - TTS/STT 우선 공급자
  - 발음 가이드 방식
  - RTL 여부
- `server/config/languages.ts` 같은 서버 대응 설정 추가
- 공용 타입 `LanguageCode`, `DifficultyFramework`, `RomanizationMode` 정의

### 산출물
- 언어 설정 테이블
- 프론트/백 공용 타입

### 리스크
- 프론트/서버 언어 코드 불일치 가능성
- shared schema에 일부 타입 승격 필요

---

## 1-3. 시나리오 구조 재설계

### 작업 목표
시나리오를 일본어 콘텐츠 묶음이 아니라 **언어 독립형 학습 장면**으로 바꾼다.

### 세부 작업
- 현재 시나리오를 2층 구조로 분리
  - 공통 시나리오 키: `cafeteria`, `travel_checkin`, `meeting_opener`
  - 언어별 콘텐츠 팩: 제목/설명/표현/레벨/톤/예시
- 기존 `ScenarioConfig`를 아래처럼 분리
  - `ScenarioCore`
  - `ScenarioLocaleContent`
  - `ScenarioLanguagePedagogy`
- 레벨 체계를 언어별로 표현 가능하게 변경
- `expressions`도 언어별 데이터로 이동
- 홈 추천/빠른 입장도 언어별 추천 규칙 지원

### 산출물
- 공통 시나리오 키 목록
- 언어별 시나리오 콘텐츠 구조
- 데이터 마이그레이션 초안

### 리스크
- 기존 페이지들이 단일 `SCENARIO_CONFIGS` 의존 중이라 대량 수정 필요
- 초기에 데이터 중복이 늘어날 수 있음

---

## 1-4. UI 국제화(i18n) 기반 도입

### 작업 목표
홈/버튼/메시지/배지/경고문을 언어에 따라 바뀌게 만든다.

### 세부 작업
- i18n 라이브러리 도입 여부 결정
  - 경량 구조면 자체 구현 가능
  - 규모 확장 고려 시 i18n 라이브러리 권장
- UI 텍스트를 리소스 파일로 분리
  - `ko`, `en`, `ja` 우선
- 페이지별 카피 키 정리
  - Landing
  - Home
  - Scenario
  - Character
  - Playground
  - Subscription
- 버튼/토스트/에러 메시지 번역 키화
- `uiLanguage` 기반 렌더링 적용
- 아랍어 대비 RTL 토글 구조 설계

### 산출물
- i18n 디렉터리 구조
- 기본 번역 리소스
- 텍스트 추출 규칙

### 리스크
- 번역 키 남용 시 관리 어려움
- 서버 에러 메시지와 프론트 문구 톤 불일치 가능

---

## 1-5. 프롬프트 엔진 분리

### 작업 목표
현재의 일본어 전용 프롬프트를 언어별 템플릿 체계로 분리한다.

### 세부 작업
- `generateDialogue()` 내부 하드코딩 로직을 모듈 분리
- 다음 계층으로 나누기
  - 공통 prompt builder
  - 언어별 pedagogical rules
  - 시나리오별 opening strategy
  - 피드백 템플릿
- 입력 스키마 확장
  - `targetLanguage`
  - `supportLanguage`
  - `difficultyFramework`
  - `difficultyLevel`
- 출력 스키마 확장
  - 원문
  - 번역
  - 설명
  - 발음 가이드
- 언어별 금지 규칙/권장 규칙 분리

### 산출물
- 언어별 prompt preset 설계
- 공통 prompt builder 함수
- 테스트 가능한 샘플 프롬프트 세트

### 리스크
- 프롬프트 복잡도가 증가하여 유지보수 난이도 상승
- 언어별 품질 차이 발생 가능

---

## 1-6. TTS/STT 계층 추상화

### 작업 목표
현재의 “일본어면 A, 아니면 B” 수준 분기를 언어별 음성 전략으로 교체한다.

### 세부 작업
- 언어별 TTS voice map 설계
- 공급자 fallback 전략 정의
- 발음 속도/톤 기본값을 언어별로 분리
- STT 지원 여부와 품질 등급 표준화
- 언어 코드 기반 보이스 선택 로직 구현
- RTL 언어 자막 표시 검증 포인트 정의

### 산출물
- `voiceProfile` 규격
- 언어별 TTS 설정표
- 공급자 우선순위 정책

### 리스크
- 언어별 음성 품질 편차 큼
- 일부 언어는 STT/TTS 조합 최적화가 추가로 필요

---

## 1-7. 평가/피드백 모델 추상화

### 작업 목표
현재의 단순 일본어 문자 기반 평가에서 언어별 피드백 엔진으로 가는 기반을 만든다.

### 세부 작업
- `feedbackMode` 개념 도입
  - deterministic-lite
  - AI-assisted
  - hybrid
- 언어별 검사 포인트 정의
  - 일본어: 경어/조사/자연스러움
  - 영어: 문법/콜로케이션/발음
  - 중국어: 어순/성조 가이드
  - 아랍어: 문자 방향/형태 변화 검토
- 공통 피드백 스키마 설계
  - accuracy
  - naturalness
  - correction
  - betterExpression
  - supportExplanation
  - pronunciationTip

### 산출물
- 다국어 피드백 데이터 계약서
- 언어별 피드백 룰 초안

### 리스크
- 언어별 채점 기준 통일이 어려움
- 초기에는 품질 불균형 가능

---

## 1-8. 데이터/스토리지 마이그레이션

### 작업 목표
기존 사용자 데이터와 새 언어 구조를 안전하게 연결한다.

### 세부 작업
- persisted store migration 버전 추가
- DB에 언어 관련 컬럼 추가 검토
- 대화 세션/복습 표현/즐겨찾기 데이터에 언어 태그 저장
- 기존 일본어 세션 기본값 자동 보정

### 산출물
- 마이그레이션 계획서
- 하위 호환 전략

### 리스크
- 이전 사용자 세션이 깨질 가능성
- 무료/유료 사용량 집계에 영향 가능

---

## 1단계 QA 체크리스트
- 일본어 기존 핵심 플로우가 유지되는가
- 언어 상태 변경 시 앱이 정상 전환되는가
- 지원 언어/학습 언어/UI 언어가 서로 충돌하지 않는가
- 시나리오/홈/플레이그라운드가 언어 메타데이터를 읽는가
- TTS가 언어 설정에 맞는 보이스를 선택하는가
- 저장 데이터가 이전 버전과 호환되는가

# 최종 제안
가장 안전한 전략은 다음과 같다.

- **지금은 1단계 엔진 추상화만 확정한다.**
- 일본어 전용 구조를 공통 코어 구조로 먼저 바꾼다.
- 그 다음 실제 코드 작업에서 일본어/영어 구현을 붙인다.

이 방식이면 설계와 구현 범위를 섞지 않고, 가장 먼저 필요한 구조 개편에 집중할 수 있다.

# 1단계 마감 운영안 / 2단계 착수 규칙

## 1단계 마감 처리 원칙
- 1단계는 **개발 완료**와 **런타임 QA 완료**를 구분해서 관리한다.
- 로컬 정적 검증/빌드/테스트가 끝났고 구조 목표가 충족되면 상태를 다음 둘 중 하나로 표기한다.
  - `1단계 완료`: 구현 + 주요 QA까지 종료
  - `1단계 구현 완료, 런타임 QA pending`: 개발은 끝났지만 실행 환경 검증만 남은 상태
- 두 번째 상태여도 2단계 계획 수립과 선행 설계는 바로 시작할 수 있다.

## 에이전트가 직접 끝까지 수행할 1단계 QA 범위
현재 이 저장소에서 작업 중인 에이전트는 아래 항목을 가능한 범위까지 **직접 연속 실행**한다.
1. 언어 선택 변경 시 홈/시나리오/플레이그라운드 문구 전환 확인
2. target/support/ui language 조합 충돌 여부 확인
3. 일본어 기존 학습 플로우 회귀 확인
4. 음성/녹음/프롬프트 호출 시 언어 메타데이터 전달 여부 확인
5. persisted store 마이그레이션 및 기존 사용자 기본값 보정 확인
6. 정책 페이지/구독/인증 화면 다국어 카피 누락 확인

### QA 결과 보고 형식
- `완료`: 에이전트가 직접 확인함
- `차단`: 환경 제약으로 실행 불가
- `후속 필요`: 수동 검증 또는 운영 환경 확인 필요

## 2단계 착수 방식
1단계가 `구현 완료, 런타임 QA pending` 상태여도 아래 작업은 즉시 진행한다.
- 2단계 목표/범위 문서화
- 구현 배치(batch) 분해
- 선행 리스크 정리
- 필요한 데이터/번역/콘텐츠 준비물 정리
- QA 자동화 후보 선정

## 2단계 계획 산출물 요구사항
2단계 계획은 최소 아래 항목을 한 번에 포함해야 한다.
- 목표와 성공 기준
- 세부 작업 묶음(batch)과 권장 순서
- 선행 의존성 / 병렬 가능 작업
- QA 전략
- 중간 보고 없이도 계속 진행 가능한 범위와, 꼭 사람 판단이 필요한 분기점


## 2단계 원샷 실행 계획
아래 순서는 사용자의 추가 승인 없이 **한 번에 계속 진행**하는 기본 시나리오다.

### Step 1. 언어팩 골격 확정
- `uiCopy`, `scenarioContent`, prompt rule, voice policy의 파일 경계를 정리한다.
- 500줄 안팎 원칙을 넘는 파일은 역할별로 쪼갤 후보를 먼저 표시한다.
- 산출물: 언어팩 구조 초안 + 파일 분리 계획

### Step 2. 영어 우선 핵심 흐름 완성
- 일본어 기준 흐름과 동일한 수준으로 영어 target language 핵심 시나리오를 채운다.
- 홈 → 시나리오 → 캐릭터 → 플레이그라운드 진입 흐름에서 영어 누락 카피를 정리한다.
- 산출물: 영어 1차 usable flow

### Step 3. support/ui language 정합성 보강
- 한국어/영어 UI copy 누락을 정리한다.
- supportLanguage에 따라 설명/에러/가이드 문구가 맞게 바뀌는지 맞춘다.
- 산출물: support/ui language 동작 일관성 확보

### Step 4. prompt/feedback/audio 품질 보정
- 영어/일본어의 교정 톤, 칭찬 톤, 설명 톤을 분리한다.
- TTS/STT 선택과 에러 fallback을 언어 기준으로 정리한다.
- 산출물: 언어별 품질 룰 정리

### Step 5. 회귀 QA + 잔여 TODO 정리
- 가능한 로컬 검증을 끝까지 수행한다.
- 환경 제약으로 못 한 항목만 별도 묶음으로 남긴다.
- 산출물: 완료/차단/후속 필요 목록 + 다음 배치 제안

# 2단계 — 일본어 + 영어 먼저 완성

## 목표
- 1단계에서 만든 다국어 코어 위에 **실제 출시 가능한 언어팩 흐름**을 얹는다.
- 최소 범위는 **일본어 안정화 + 영어 1차 지원 + 한국어 UI/support 정합성 확보**다.
- 사용자가 언어를 바꿨을 때, 단순 라벨 변경이 아니라 **시나리오/프롬프트/피드백/음성 경험 전체**가 자연스럽게 이어져야 한다.

## 성공 기준
- 일본어 학습 플로우가 새 코어 위에서 회귀 없이 동작한다.
- 영어를 target language로 선택했을 때 핵심 시나리오/프롬프트/피드백/TTS 흐름이 끊기지 않는다.
- 한국어와 영어 UI 카피가 주요 진입 화면에서 빠짐없이 제공된다.
- 운영자가 새 언어를 넣을 때 수정 지점이 문서화되고, 언어팩 추가 절차가 반복 가능해야 한다.

## 작업 묶음(Batch)

### Batch 2A — 언어팩 최소 단위 확정
- 언어팩 디렉터리/파일 규칙 확정
- 언어별 필수 리소스 목록 정의
  - UI copy
  - scenario locale content
  - pedagogical rules
  - TTS/STT defaults
  - feedback tone / explanation templates
- 일본어/영어 기준 샘플 언어팩 구조 완성
- 한국어는 support/UI 우선 팩으로 정리

### Batch 2B — 핵심 시나리오 언어팩 채우기
- 우선 시나리오 5~10개 선정
  - 자기소개
  - 카페/주문
  - 길 묻기
  - 여행 체크인
  - 회의 시작 / 비즈니스 스몰토크
- 각 시나리오에 대해 일본어/영어 콘텐츠 품질 보강
- 표현/톤/난이도/학습 포인트를 언어별로 분리
- 홈 추천 규칙과 빠른 시작 진입도 언어별로 연결

### Batch 2C — 프롬프트/피드백 품질 고도화
- 영어용 pedagogical rule set 정교화
- 일본어/영어별 opening strategy, correction style, praise style 분리
- supportLanguage별 explanation 템플릿 보강
- 응답 JSON 계약 검증과 fallback 문구 체계 정리

### Batch 2D — 음성/입출력 품질 정리
- 언어별 TTS voice selection 실제 점검
- STT 입력 언어와 targetLanguage 연결 검증
- 발음 가이드 표기 규칙 정리
- 녹음 실패/브라우저 미지원 시 언어별 에러 UX 정리

### Batch 2E — 화면 마감 및 운영 준비
- Landing/Home/Scenario/Character/Playground/Auth/Subscription의 카피 누락 정리
- 정책 페이지 다국어 톤 일관성 점검
- 운영/분석 포인트 정의
  - 어떤 언어 조합이 많이 쓰이는지
  - 어떤 언어에서 이탈이 큰지
  - TTS/STT 실패율이 언어별로 어떤지
- 신규 언어 추가 가이드 문서 초안 작성

## 권장 실행 순서
1. Batch 2A — 언어팩 최소 단위 확정
2. Batch 2B — 핵심 시나리오 언어팩 채우기
3. Batch 2C — 프롬프트/피드백 품질 고도화
4. Batch 2D — 음성/입출력 품질 정리
5. Batch 2E — 화면 마감 및 운영 준비

## 병렬 가능 작업
- 2A와 동시에 UI 카피 누락 점검 시작 가능
- 2B와 2C는 시나리오 선정만 끝나면 병렬 진행 가능
- 2D는 TTS/STT 설정표가 있으면 독립 검증 가능
- 2E의 운영 문서는 앞 배치 진행 중에도 병행 작성 가능

## 에이전트 연속 실행 지침
- 에이전트는 2A만 해놓고 멈추지 말고, 치명적 블로커가 없으면 2B·2C·2D·2E까지 순차 진행한다.
- 즉, 사용자가 따로 “계속해”를 반복 입력하지 않아도 되도록 **2단계를 하나의 실행 묶음**으로 본다.
- 각 Batch 종료 시점마다 다시 허락을 기다리지 말고, **치명적 블로커가 없으면 다음 Batch 구현 또는 초안 작성**까지 이어서 수행한다.
- 보고는 “완료된 Batch / 진행 중 Batch / 차단 요소 / 다음 자동 착수 항목” 형식으로 묶어서 한 번에 한다.

## QA 전략
- 정적 검증: typecheck, build, schema consistency 확인
- 기능 검증: 언어 변경 → 시나리오 진입 → 대화 생성 → 음성 재생 흐름 확인
- 회귀 검증: 일본어 기존 핵심 경로 우선 확인
- 카피 검증: 주요 페이지 locale key 누락/영문 fallback 누락 확인
- 운영 검증: 실패 로그에 target/support/ui language가 남는지 확인

## 사람 판단이 꼭 필요한 분기점
- 출시 우선순위를 일본어+영어 2개 언어로 고정할지 여부
- 영어 피드백의 엄격도(학습 친화형 vs 시험 대비형)
- 한국어를 UI 전용으로 둘지, support language까지 1차 출시 범위에 넣을지 여부
- 언어별 TTS 공급자 비용/품질 트레이드오프 승인

## 중간 승인 없이 계속 진행 가능한 범위
- 언어팩 파일 구조 정리
- 기존 시나리오의 locale content 분리/보강
- UI 카피 누락 보완
- prompt template / feedback template 정리
- QA 매트릭스 작성 및 자동화 후보 분류

# 3단계 — 나머지 언어 확장

## 목표
- 1단계 공통 엔진 + 2단계 일본어/영어 검증 결과를 바탕으로 나머지 언어를 **언어 패키지 추가 방식**으로 확장한다.
- 새 언어를 넣을 때 코어 로직을 크게 다시 건드리지 않는 구조를 유지한다.

## 우선 확장 후보
- 한국어
- 중국어
- 프랑스어
- 독일어
- 스페인어
- 베트남어
- 태국어
- 아랍어

## 확장 원칙
- 각 언어는 최소 아래 패키지를 가진다.
  - 표시명
  - 난이도 체계
  - TTS/STT 설정
  - 프롬프트 템플릿
  - 교정 규칙
  - 대표 시나리오 표현
  - 발음/문법 피드백 규칙
- 아랍어는 RTL UI/텍스트 정렬 검증까지 포함하므로 마지막 확장군으로 두는 것이 안전하다.

## 완료 기준
- 새 언어 추가가 코어 기능 수정이 아니라 패키지 추가/등록 작업으로 끝나야 한다.
- 언어별 품질 편차를 QA 매트릭스로 추적할 수 있어야 한다.
